# 知宠服务器 AI 使用宪法

版本：1.0  
适用范围：知宠团队管理的开发、预发布和生产服务器，以及通过团队账号、自动化任务或远程终端执行操作的任何 AI 系统。

## 一、效力与边界

1. 本宪法约束 AI、调用 AI 的人员和由 AI 生成后交由自动化执行的命令。
2. AI 在服务器上的默认权限是观察、分析和提出方案，不是自由修改。
3. 文档约束不能替代操作系统权限。生产保护必须同时依靠独立账号、最小权限、受控部署、备份和日志。
4. 人与 AI 不得共用同一把 SSH 私钥。AI 必须使用可单独吊销、可追踪的身份进入服务器。
5. 当本宪法与项目内更严格的 `AGENTS.md` 冲突时，执行更严格的规则。

## 二、不可突破的原则

1. 不得编造服务器状态、部署结果、备份结果、监控结果或安全结论。
2. 不得读取、输出、复制或上传与当前任务无关的密钥、授权码、数据库连接串、用户数据和私人目录内容。
3. 不得把 `.env`、数据库导出、日志中的个人信息、SSH 私钥或云平台密钥提交到 Git。
4. 不得直接修改生产发布目录中的构建产物；生产变更必须来自已审核的 Git 提交。
5. 不得绕过测试、审批、备份、健康检查或回滚步骤来追求“尽快上线”。
6. 不得以 Docker 组、sudo、root shell 或云平台管理员权限作为日常开发权限。
7. 不得对外开放数据库、Nuxt 内部端口、管理端口或临时调试服务。
8. 不得在未经确认的情况下更改域名、DNS、证书、备案信息、用户、用户组、SSH、安全组、防火墙和数据库结构。

## 三、默认操作规则

AI 每次进入服务器后必须按以下顺序工作：

1. 读取本宪法和当前项目的 `AGENTS.md`。
2. 输出当前身份、主机名、工作目录、Git 分支和工作区状态。
3. 明确本次任务、允许修改的路径、需要使用的权限和验证方式。
4. 先执行只读检查，再提出或执行最小范围修改。
5. 修改前说明影响、回滚方法和是否涉及服务中断。
6. 修改后执行项目规定的检查，并报告真实输出。
7. 保留操作记录；出现异常时立即停止继续扩大变更。

未得到明确授权时，AI 只能执行以下类型的操作：

- 查看状态、日志、文件元数据和公开配置。
- 在个人家目录或明确指定的开发工作区内创建临时文件。
- 运行不会改变生产数据的 lint、typecheck、test 和 build。
- 提交变更方案、差异和回滚计划供人工审核。

## 四、高风险操作

以下操作必须同时具备明确的人类批准、变更说明、备份或回滚方案：

- `sudo`、root、Docker daemon 或云平台管理操作。
- 安装或移除系统软件包。
- 修改 systemd、Nginx、SSH、防火墙、安全组、定时任务和证书。
- 创建、删除或调整用户、用户组和目录权限。
- 数据库迁移、批量更新、删除、恢复和导入导出。
- 启停生产服务、切换发布版本和开放公网流量。
- 递归删除、递归移动、覆盖式复制或强制 Git 操作。

禁止把以下命令作为试探手段：

- `rm -rf`、`git reset --hard`、`git clean -fdx`。
- `DROP DATABASE`、`DROP TABLE`、无条件批量 `DELETE`。
- 未核实目标路径的 `chmod -R`、`chown -R`、`mv` 和 `rsync --delete`。
- 把远程脚本直接通过管道交给 root shell 执行。

## 五、路径与所有权

| 区域 | 用途 | AI 默认权限 |
| --- | --- | --- |
| `/home/<user>` | 个人文件 | 仅对应身份可写，不读取他人内容 |
| `/srv/devteam` | 团队公共代码 | 团队开发可写，必须保留 setgid 和 Git 记录 |
| `/srv/zhipet/releases` | 不可变发布版本 | 只允许部署账号写入 |
| `/srv/zhipet/current` | 当前版本软链接 | 只允许受控部署切换 |
| `/etc/zhipet` | 生产环境变量 | root 可写，运行账号只读必要内容 |
| `/var/log/zhipet` | 应用日志 | 应用写入，成员按需只读 |

任何 AI 都不得把个人实验文件、下载包或数据库备份放入生产目录。

## 六、Git 与发布

1. 开发变更从功能分支进入，禁止在服务器上直接修改 `main` 后发布。
2. 发布来源必须是远程仓库中可定位的提交哈希。
3. 发布前至少通过 `pnpm lint`、`pnpm typecheck`、`pnpm test` 和 `pnpm build`。
4. 数据库迁移只能使用 `prisma migrate deploy`，生产环境禁止使用 `migrate dev`。
5. 发布采用新目录构建、健康检查、原子切换和保留上一版本的方式。
6. 发布失败必须停止切换并保留现场，不得连续尝试未经分析的修复命令。

## 七、数据与隐私

1. 联系方式、反馈正文、宠物数据、家庭环境数据和原始音频均按敏感数据处理。
2. 日志不得记录请求正文、Cookie、SMTP 密钥、数据库连接串和完整联系方式。
3. AI 不得把生产数据用于测试、提示词、外部模型训练或第三方排障。
4. 排障需要样本时，应使用脱敏、最小化的人工构造数据。
5. 数据清理、备份和恢复必须遵守项目的数据保存规则并留下审计记录。

## 八、停止条件与事故处理

出现以下任一情况，AI 必须立即停止修改并通知负责人：

- 实际主机、账号、路径或环境与任务描述不一致。
- 发现未提交改动、未知进程、未知端口或来源不明的密钥。
- 需要超出批准范围的 sudo、数据库或网络权限。
- 测试、构建、迁移、备份或健康检查失败。
- 目标路径无法确定，或命令可能影响个人目录和生产数据。
- 发现疑似泄密、入侵、误删或权限扩大。

事故处理中不得为了隐藏错误而删除日志、重写历史或清理现场。

## 九、身份与硬约束

推荐为每条 AI 操作渠道分别创建可审计的受限身份，例如：

- `member-a-ai`
- `member-b-ai`
- `member-c-ai`
- `member-d-ai`

这些身份不得加入 `sudo`、`docker` 或 `lxd` 组。确需发布时，只能调用经过审核、参数固定并写入日志的部署命令。团队成员的人类账号和 AI 身份应使用不同 SSH 密钥。

若 AI 继续使用团队成员的人类账号，且这些账号拥有 sudo 或 Docker 权限，则本宪法只能形成行为约束，不能形成安全隔离。

## 十、修订

1. 本宪法的修改必须进入 Git 评审，不得只在服务器上临时改写。
2. 放宽权限需要项目负责人明确批准；收紧权限可作为安全修复提交。
3. 服务器上的生效副本必须可追溯到仓库版本，并通过审计脚本检查一致性。
